# 食谱 RAG 系统 — 功能实现说明 v1.0

## 1. 项目概述

基于 **检索增强生成（Retrieval-Augmented Generation, RAG）** 架构的智能食谱问答系统。  
数据源为 [HowToCook](https://github.com/Anduin2017/HowToCook) 开源食谱库（Markdown 格式），覆盖中式家常菜、汤羹、早餐、甜品等十余个品类。

### 1.1 系统流程

```
用户查询 → 查询改写 → 混合检索 → 意图路由 → 模板渲染 → 回答
```

### 1.2 技术栈

| 组件 | 技术选型 |
|------|----------|
| 向量检索 | FAISS (IndexFlatIP + L2 归一化) |
| 稀疏检索 | BM25 (rank_bm25) |
| 中文分词 | jieba |
| 文本嵌入 | sentence-transformers (shibing624/text2vec-base-chinese) |
| 编程语言 | Python 3.10+ |

## 2. 数据预处理（Preprocess）

### 2.1 模块结构

```
src/preprocess/
├── config.py      # 全局配置（路径、模型名、分词器）
├── splitter.py    # RecipeSplitter — Markdown 层级分块
└── indexer.py     # 索引构建入口（FAISS + BM25）
```

### 2.2 层级分块策略（`RecipeSplitter`）

每个 `.md` 食谱文件被解析为**三级层级结构**：

| 层级 | 名称 | 来源 | 示例 |
|------|------|------|------|
| L1 | dish（菜品级） | `# H1` + H1 到第一个 `##` 之间的前言 | 整道菜概览，含难度、卡路里 |
| L2 | section（章节级） | 每个 `##` 小节 | `## 操作`、`## 必备原料和工具` |
| L3 | subsection（子步骤级） | 仅在 `## 操作` 下的 `###` | 预处理、主步骤、收尾等 |

**分块处理细节：**

- **Footer 移除**：自动去除 GitHub PR 模板脚注行
- **操作步骤展开**：`SPLIT_H3 = True` 时，`## 操作` 内的 `###` 子标题会被拆为独立 L3 块，便于精确检索具体步骤
- **元数据提取**：从 H1 标题自动去除 "的做法" 后缀得到 dish_name；从前言正则匹配 difficulty（预估烹饪难度）和 calories（预估卡路里）
- **分类标签**：根据文件在 `dishes/` 下的父目录名确定 category

### 2.3 索引构建（`build_index`）

索引构建分三步执行：

```
collect_all_recipes()       # 遍历目录收集所有 .md 文件路径
       ↓
RecipeSplitter.split()      # 逐文件分块，产生 N 个 chunk
       ↓
编码 → FAISS IndexFlatIP   # 用 text2vec-base-chinese 生成 768 维向量，L2 归一化后建索引
       ↓
分词 → BM25Okapi           # jieba 精确模式分词，建稀疏索引
```

**输出文件（`data/vectorstore/`）：**

| 文件 | 格式 | 内容 |
|------|------|------|
| `faiss.index` | FAISS 二进制 | 稠密向量索引 |
| `chunks.pkl` | pickle | chunk 文本 + 元数据列表 |
| `bm25_index.pkl` | pickle | BM25 模型 + 分词语料库 |

**缓存机制**：索引加载后缓存在模块级字典 `_RETRIEVAL_CACHE` 中，避免重复读取。

## 3. 检索模块（Retrieval）

### 3.1 模块结构

```
src/retrieval/
├── hybrid.py      # 混合检索（稠密 + 稀疏 + RRF 融合 + 推荐）
├── filters.py     # 元数据过滤（难度、类别、卡路里、菜名）
└── diversity.py   # 多样性重排序（类别轮询 + MMR）
```

### 3.2 稠密检索（`dense_search`）

- 使用 `text2vec-base-chinese` 将查询编码为 768 维向量
- 在 FAISS IndexFlatIP（内积索引）中执行 top-k 搜索
- 已做 L2 归一化，等价于余弦相似度

### 3.3 稀疏检索（`sparse_search`）

- 使用 jieba 对查询做中文分词
- 通过 BM25Okapi 计算词袋相似度
- 过滤得分为 0 的结果

### 3.4 混合检索与 RRF 融合（`hybrid_search`）

**算法：倒数排名融合（Reciprocal Rank Fusion, RRF）**

```
RRF_score(chunk) = 1 / (k + rank_dense) + 1 / (k + rank_sparse)
```

- `k = 60.0`（经典值）
- 两个通道各取 top-50 候选，合并后按 RRF 得分排序
- 返回每个 chunk 的 RRF 分数、稠密排名、稀疏排名

### 3.5 菜品推荐（`recommend_dishes`）

面向推荐场景的专用检索流程：

```
多探针搜索 (multi-probe) → 元数据过滤 → 多样性重排序
```

**多探针搜索**：接受 `probes` 参数（来自查询改写模块），对每个改写查询分别做 hybrid_search，合并去重后只保留 dish 级（L1）chunk，最多使用 3 个探针。

**过滤器（`filters.py`）：**

| 过滤字段 | 匹配方式 | 示例值 |
|----------|----------|--------|
| `difficulty` | 前缀匹配 | `★`、`★★` |
| `category` | 精确匹配 | 素菜、肉菜、汤 |
| `calories` | 数值阈值 | `low`(≤300)、`high`(≥600) |
| `level` | 精确匹配 | dish、section、subsection |
| `target_dish` | 菜名字符串子串匹配（大小写不敏感） | 红烧肉 |

**多样性重排序（`diversity.py`）：**

提供两种策略：

1. **MMR（最大边界相关）**：在查询相关性与已选结果多样性之间平衡，`lambda` 参数控制权重（0.5 默认）
2. **类别轮询（Round-robin）**：按 category 分组后轮流选取，保证结果覆盖不同品类

当 MMR 可用时优先使用，异常时降级为类别轮询。

### 3.6 One-Shot 搜索工具

`hybrid_search` 提供无状态单次搜索接口，用于 `howto`、`ingredient` 等意图的直接检索。

## 4. 查询改写（Rewriting）

### 4.1 模块结构

```
src/rewriting/
├── intent.py      # 意图分类 + 约束提取（规则驱动）
└── rewriter.py    # 改写器抽象 + 规则实现 + LLM 扩展接口
```

### 4.2 意图分类（`classify_intent`）

纯规则驱动，无外部依赖。支持四种意图：

| 意图 | 触发关键词 | 路由目标 |
|------|-----------|----------|
| `recommendation` | 今天吃什么、推荐、来点、下饭菜… | `recommend_dishes` |
| `howto` | 怎么做、做法、步骤、怎么烧… | `hybrid_search`（过滤操作章节） |
| `ingredient` | 材料、原料、食材、配料… | `hybrid_search`（过滤原料章节） |
| `factual` | 是什么、哪里的菜、区别… | `hybrid_search`（通用） |

匹配规则采用 **最长关键词优先** 策略。

### 4.3 约束提取

从查询中正则匹配并提取结构化过滤条件：

- **难度**：简单→★，中等→★★，困难→★★★★
- **类别**：素菜、肉菜、汤、早餐、水产、主食、甜点
- **卡路里**：低热量/减肥→`low`，高热量/硬菜→`high`
- **目标菜名**：通过 `_DISH_PATTERN` 去除意图后缀后的剩余文本

### 4.4 查询改写

- **推荐意图**：去除 "推荐/今天/吃什么" 等无意义词，保留食物相关术语；无剩余时默认 "家常菜"
- **做法意图**：拼接为 "菜名 操作 步骤"
- **原料意图**：拼接为 "菜名 必备原料和工具"

### 4.5 多探针生成（`_build_probes`）

为推荐意图生成多个搜索探针以增加召回多样性：

1. 改写后的主查询
2. 类别限定查询（如有类别过滤条件）
3. "家常菜" / "下饭" 变体
4. 难度限定查询（如为简单难度）

最多生成 5 个探针。

### 4.6 改写器架构

```
BaseQueryRewriter（抽象基类）
├── RuleQueryRewriter   # 当前默认实现，零外部依赖
└── LLMQueryRewriter    # 预留 LLM 扩展接口，当前降级为规则
```

## 5. 生成模块（Generation）

### 5.1 模块结构

```
src/generation/
├── base.py         # Generator 抽象基类
├── template.py     # TemplateGenerator — 基于模板的渲染
└── pipeline.py     # RAGPipeline — 端到端流程编排
```

### 5.2 生成器抽象

`Generator` 抽象基类定义 `generate(query, context, intent) -> str` 接口，支持替换为 LLM 实现。

### 5.3 模板生成器（`TemplateGenerator`）

按意图渲染结构化文本：

- **recommendation**：编号列表，每项含菜名、类别、难度、卡路里
- **howto**：按菜名分组，展示完整操作文本
- **ingredient**：逐菜展示原料清单
- **其他**：显示相关片段预览（前 150 字）

空结果返回友好提示（中文）。

### 5.4 `RAGPipeline` — 端到端编排

```
run(query):
  1. rewrite → IntentResult（意图 + 改写 + 探针 + 过滤器）
  2. retrieve → context（按意图路由到不同检索策略）
  3. generate → answer（模板渲染）
```

**意图路由矩阵：**

| 意图 | 检索方法 | 后处理 |
|------|---------|--------|
| recommendation | `recommend_dishes`（多探针 + 过滤 + 多样性） | — |
| howto | `hybrid_search`（k=50） | 优先返回匹配菜名的 `## 操作` 章节块 |
| ingredient | `hybrid_search`（k=50） | 优先返回匹配菜名的 `## 必备原料和工具` 章节块 |
| fallback | `hybrid_search`（top_k） | — |

`trace()` 方法提供调试视图，输出完整中间状态（意图、改写、探针、过滤器、命中块明细）。

## 6. 运行说明

### 6.1 环境配置

```bash
pip install -r requirements.txt
```

依赖：`faiss-cpu`、`sentence-transformers`、`numpy`、`rank-bm25`、`jieba`。

### 6.2 构建索引

```bash
python -m src.preprocess.indexer
```

首次运行会自动下载 `shibing624/text2vec-base-chinese` 模型。输出见 `data/vectorstore/`。

### 6.3 测试检索

```python
from src.retrieval import hybrid_search, recommend_dishes

# 混合搜索
results = hybrid_search("红烧肉 做法", k=5)

# 菜品推荐
recommendations = recommend_dishes(
    query="今天想吃点清淡的",
    k=3,
    filters={"calories": "low"},
)
```

### 6.4 使用 Pipeline

```python
from src.generation import RAGPipeline

pipeline = RAGPipeline()
answer = pipeline.run("麻婆豆腐怎么做")
print(answer)

# 查看调试详情
trace = pipeline.trace("今天吃什么")
print(trace["intent"], trace["filters"])
```

## 附录 B：关键设计决策

1. **层级分块而非固定窗口**：保留 Markdown 结构，支持按章节粒度检索，提高步骤级查询的命中精度
2. **RRF 融合而非加权平均**：无需调参，对稠密/稀疏通道的分数尺度不敏感
3. **规则驱动改写（v1）**：零外部依赖，毫秒级延迟；预留 LLM 接口便于后续升级
4. **多探针 + 多样性重排序**：针对推荐场景避免结果同质化，提升用户覆盖度
5. **TemplateGenerator 先行**：流水线开发阶段可独立调试检索质量，后续替换为 LLM Generator 无需改动路由逻辑
