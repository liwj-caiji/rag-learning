## 4.7（v1.1 新增）LLM 意图识别

### 4.7.1 概述

`LLMIntentClassifier` 使用 **deepseek-v4-flash** 模型替代纯规则匹配，通过结构化输出（JSON mode）同时完成意图分类、约束提取、查询改写和多探针生成。

API 配置：

| 参数 | 默认值 |
|------|--------|
| Model | `deepseek-v4-flash` |
| API Base | `https://api.deepseek.com` |
| Auth | 环境变量 `DEEPSEEK_API_KEY` |

### 4.7.2 工作流程

```
LLMIntentClassifier.classify(query)
  ├── API key 存在？ → 调用 LLM（JSON mode, temperature=0.1, max_tokens=512）
  │     ├── 成功 → 解析 JSON → IntentResult
  │     └── 失败 → 降级到规则分类
  └── API key 不存在 → 降级到规则分类
```

### 4.7.3 LLM 提示词设计

**System Prompt** 定义：
- 四种意图的分类标准（recommendation / howto / ingredient / factual）
- JSON 输出结构 schema
- 约束提取规则（难度等级映射、类别推断、卡路里阈值）

**LLM 输出 JSON 示例：**

```json
{
  "intent": "recommendation",
  "rewritten_query": "清淡 家常菜",
  "filters": {
    "difficulty": "★",
    "category": "素菜",
    "calories": "low"
  },
  "target_dish": null,
  "probes": ["清淡素菜", "家常素菜", "快手素菜"],
  "confidence": 0.95
}
```

### 4.7.4 降级策略

LLM 调用失败（网络超时、API 不可用、JSON 解析错误）时自动降级为规则分类器 `classify_intent`，保证服务可用性。

### 4.7.5 与规则分类对比

| 维度 | 规则分类 | LLM 分类 |
|------|---------|---------|
| 依赖 | 无 | DEEPSEEK_API_KEY |
| 延迟 | <1ms | ~500-2000ms |
| 泛化能力 | 需手动添加关键词 | 自然语言理解 |
| 约束提取 | 正则匹配 | 语义推断 |
| 探针质量 | 模板拼接 | 上下文感知生成 |

## 5.4（v1.1 新增）LLM 生成器

### 5.4.1 概述

`LLMGenerator` 使用 deepseek-v4-flash 根据检索结果生成自然语言回答。  
实现了 `Generator` 抽象基类，与模板生成器可互换。

API 配置与 LLMIntentClassifier 一致（deepseek-v4-flash / DEEPSEEK_API_KEY）。

### 5.4.2 提示词设计

按意图类型定制 system prompt：

| 意图 | 指令重点 |
|------|---------|
| recommendation | 给出推荐理由，列出菜名、类别、难度、卡路里 |
| howto | 按顺序整理操作步骤，保留火候/时间/用量细节 |
| ingredient | 整理必备原料清单，标注用量 |
| factual | 基于上下文直接回答知识性问题 |

**通用约束**（所有意图共用）：
- 只使用检索到的上下文回答，不编造菜谱
- 信息不足时如实告知
- 结构清晰、中文回答
- 可引用菜名作为来源

### 5.4.3 上下文格式化

检索到的 chunks 被格式化为结构化文本供 LLM 参考：

```
[1] 麻婆豆腐 - 必备原料和工具
（文本内容...）
---
[2] 麻婆豆腐 - 操作（预处理）
（文本内容...）
---
[3] 麻婆豆腐 - 操作（主步骤）
（文本内容...）
```

### 5.4.4 降级策略

与 `LLMIntentClassifier` 一致：API 调用失败时降级到 `TemplateGenerator`，保证管线可用。

### 5.4.5 与模板生成对比

| 维度 | TemplateGenerator | LLMGenerator |
|------|------------------|-------------|
| 回答质量 | 固定模板，机械拼接 | 自然语言，语义连贯 |
| 上下文理解 | 无 | 能综合多段信息 |
| 依赖 | 无 | DEEPSEEK_API_KEY |
| 延迟 | <1ms | ~1-3s |
| 容错 | 始终可用 | 降级到模板 |

## 5.5（v1.1修改）RAGPipeline — 端到端编排

### 5.5.1 新增 LLM 模式

```
run(query, use_llm=False):
  1. rewrite → IntentResult（LLM 或规则，根据 use_llm 标志）
  2. retrieve → context（按意图路由到不同检索策略）
  3. generate → answer（LLM 或模板，根据 use_llm 标志）
```

### 5.5.2 LLM 模式激活条件

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `use_llm` | `False` | 设为 `True` 启用 LLM 组件 |
| `llm_model` | `deepseek-v4-flash` | 模型名称 |
| `llm_api_base` | `https://api.deepseek.com` | API 地址 |

`use_llm=True` 时自动替换：

| 位置 | v1.0 | v1.1 |
|------|------|------|
| 改写器 | `RuleQueryRewriter` | `LLMIntentClassifier` |
| 生成器 | `TemplateGenerator` | `LLMGenerator` |

API 密钥通过环境变量 `DEEPSEEK_API_KEY` 提供；缺失时抛出 `ValueError`。

## 6.5（v1.1新增）使用 LLM Pipeline

```python
from src.generation import RAGPipeline

# LLM 模式
pipeline = RAGPipeline(use_llm=True)
answer = pipeline.run("今天想吃点清淡的，有什么推荐？")
print(answer)

# 查看调试详情
trace = pipeline.trace("今天吃什么")
print(trace["intent"], trace["filters"])
```

### 6.5.1 仅使用 LLM 意图识别（不使用 LLM 生成）

```python
from src.rewriting import LLMIntentClassifier
from src.generation import RAGPipeline, TemplateGenerator

classifier = LLMIntentClassifier()
result = classifier.classify("有没有适合新手做的简单肉菜？")
print(result.intent, result.filters, result.probes)

# 混搭：LLM 意图 + 模板答案
pipeline = RAGPipeline(
    rewriter=classifier,
    generator=TemplateGenerator(),
)
```

## 附录 D（v1.1新增）：关键设计决策（v1.1 追加）

1. **规则 + LLM 双轨架构**：规则驱动零延迟可离线运行；LLM 提供更深层的语义理解，失败时降级到规则
2. **生成器策略模式**：`Generator` 抽象基类允许无缝切换模板/LLM 生成器，路由逻辑与渲染解耦
3. **LLM 采用 OpenAI 兼容协议**：`deepseek-v4-flash` 通过标准 API 调用，方便替换为其他模型（如 Claude、GPT）
